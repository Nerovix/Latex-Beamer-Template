\section{题意与建模}

\begin{frame}{1. 背景与建模对象（1.5 min）}
\textbf{场景：}校内 AI 计算中心给同学提供免费算力，调度程序要在主机上实时响应请求。\\[4pt]
\textbf{系统总资源：}
\begin{itemize}
  \item 最多同时运行 $n$ 个任务（进程位）
  \item 总 GPU 数 $K$（题面中同质显卡）
  \item 总共享内存 $M$ GB
\end{itemize}
\textbf{每个任务公开信息：}
\begin{itemize}
  \item 任务编号递增，进程编号 $0\sim n-1$
  \item 需求 $(k_i,m_i)$，每段时长 $t_i$，重要性 $W_i$
\end{itemize}
\imgplaceholder{“任务到达队列 + GPU/内存资源池”系统框图}
\end{frame}

\begin{frame}{2. 任务模型与目标函数（2 min）}
任务 $i$ 的 checkpoint 机制：
\begin{itemize}
  \item 任务开始时刻就算第一个 checkpoint
  \item 相邻 checkpoint 间隔恒为 $t_i$（整小时）
  \item 总 checkpoint 数 $c_i$ \textbf{开始未知}，仅在任务结束时获知
  \item 因题目简化，总时长严格为 $c_i\cdot t_i$
\end{itemize}
调度目标：
\[
\min \sum_i W_i \cdot \frac{T_i}{t_i\cdot c_i}
\]
其中 $T_i$ 是任务从\textbf{发出到完成}的总周转时间。\\
\textbf{直觉：}高 $W_i$ 的任务不能长期等待；同等等待下，短任务段更“敏感”。
\end{frame}

\begin{frame}{3. checkpoint 与抢占的坑点（2 min）}
\begin{itemize}
  \item 抢占本身不耗时，但会回滚到“上一个 checkpoint”。
  \item 损失的是“上个 checkpoint 到当前时刻”的整段进度。
  \item 所以越接近下个 checkpoint，被抢占越不划算。
  \item 抢占是允许的，且是应对在线未知信息的关键工具。
\end{itemize}
\vspace{4pt}
所以调度要权衡：
\begin{itemize}
  \item \textbf{保留已投入进度}：避免无谓回滚
  \item \textbf{降低高价值等待}：必要时果断抢占
\end{itemize}
\imgplaceholder{“checkpoint 进度条 + 抢占回退”时间轴图}
\end{frame}

\begin{frame}{4. 交互机制、合法性与评分（2.5 min）}
\textbf{接口：}
\begin{itemize}
  \item \texttt{Init(process\_cnt, gpu\_cnt, memory\_sum)}
  \item \texttt{Event(now\_time, event)}
  \item 可调用 \texttt{Schedule(task\_id, process\_id)} / \texttt{Appoint(appoint\_time)}
\end{itemize}
\textbf{事件类型（event\_type=1..4）：}
\begin{enumerate}
  \item 任务运行到 checkpoint
  \item 任务完全结束（此时才知道该任务真实 $c_i$）
  \item 新任务到达
  \item 你预约的调度事件触发
\end{enumerate}
\textbf{合法性红线：}
\begin{itemize}
  \item \texttt{Schedule} 时任务不能已在其他进程运行，且资源需可满足
  \item \texttt{Appoint(appoint\_time)} 必须满足 $appoint\_time\ge now\_time$
\end{itemize}
\textbf{评分：}$score=15\cdot\min(0.95\cdot std/your,1)$，20 个点，总时限 8s（C++）。
\end{frame}

\section{题解代码思路}

\begin{frame}{5. 总体策略：打分 + 选集 + 列表调度（1.5 min）}
题解核心分三步：
\begin{enumerate}
  \item 给每个未完成任务计算优先级分数 \texttt{sc}
  \item 在进程数/GPU/内存约束下贪心选出集合 $S$
  \item 对 $S$ 做 list schedule：尽量保留已有运行，必要时有代价地抢占
\end{enumerate}
\vspace{4pt}
\textbf{优点：}
\begin{itemize}
  \item 计算量可控（适配 1e4 任务）
  \item 在线可增量更新
  \item 对多资源约束有直接适配
\end{itemize}
\end{frame}

\begin{frame}{6. 数据结构设计（1.5 min）}
\texttt{Task} 维护：
\begin{itemize}
  \item 静态属性：$k,m,t,W$
  \item 动态状态：WAITING/RUNNING/FINISHED
  \item \texttt{checkpoints\_passed}、等待时间累计、绑定进程
\end{itemize}
\texttt{Process} 维护：当前任务 ID 与当前段起点时间。\\[4pt]
全局状态：\texttt{usedK/usedM}、\texttt{tasks} 哈希表、\texttt{mean\_segments}（未知段数先验）。
\imgplaceholder{“任务池 + 进程池 + 资源计数器”结构图}
\end{frame}

\begin{frame}{7. 任务打分公式（2 min）}
候选值（代码中的 \texttt{Candidate}）：
\[
\hat c_i=\max(1,\ \text{passed}+\text{mean\_segments}),\quad
\phi_i=\frac{W_i}{\hat c_i\,t_i}
\]
\[
\text{sc}_i=\frac{\phi_i}{d_i\cdot t_i^{\theta}},\quad
d_i=\alpha\frac{k_i}{K}+(1-\alpha)\frac{m_i}{M}
\]
其中 $\alpha$ 按当前资源紧缺度自适应（偏 GPU 或偏内存）。\\[4pt]
\textbf{解释：}
\begin{itemize}
  \item 权重高 + 估计总段数少 + 单段短 => 更优先
  \item 占资源更“重”的任务会被折价
\end{itemize}
\end{frame}

\begin{frame}[fragile]{8. 选集合 S：近似“多约束背包”贪心（1.5 min）}
代码先对全部任务计算分数，取前若干高分候选（\texttt{nth\_element}），再按分数降序装入资源桶：
\begin{lstlisting}
int nleft=n, kleft=K, mleft=M;
for (candidate c in sorted_candidates) {
  if (kneed<=kleft && mneed<=mleft && nleft>0) {
      choose(c);
      nleft--; kleft-=kneed; mleft-=mneed;
  }
}
\end{lstlisting}
\textbf{复杂度优势：}避免每次全量排序，适合大数据点实时调度。
\end{frame}

\begin{frame}{9. 抢占判定：收益是否覆盖损失（2 min）}
若任务 $I$ 想抢占进程上任务 $J$，代码比较：
\[
\phi_I\cdot r_J > \lambda_{preempt}\cdot \phi_J\cdot s_J
\]
\begin{itemize}
  \item $s_J$：从上次 checkpoint 跑到现在的时长（被抢占会丢）
  \item $r_J$：离下一个 checkpoint 还差多久
\end{itemize}
并计算驱逐代价：
\[
\text{cost}=\lambda_1\phi_J s_J+\lambda_2\phi_J r_J
\]
在可抢占进程中选代价最小者。
\imgplaceholder{“抢占收益 vs 回退损失”对比图}
\end{frame}

\begin{frame}{10. 事件处理细节（1.5 min）}
\texttt{Event()} 中四类更新：
\begin{itemize}
  \item type1 checkpoint：\texttt{checkpoints\_passed++}，刷新段起点
  \item type2 finish：释放资源，更新 \texttt{mean\_segments}，删任务
  \item type3 arrival：加入任务池并标记等待
  \item type4 appoint：仅触发一次调度轮
\end{itemize}
然后按条件触发重调度（前期事件密集全调度，后期抽样调度）。
\end{frame}

\section{正确性、复杂度与工程}

\begin{frame}{11. 合法性保证（1.5 min）}
本题 0 分风险在“非法调用”。题解通过多层检查规避：
\begin{itemize}
  \item \texttt{task\_process\_check}: 校验资源是否可满足（含抢占后）
  \item 仅对 WAITING 任务做启动，不会一任务多进程并跑
  \item \texttt{process\_id} 范围由进程数组控制
  \item \texttt{Appoint} 只会传 $\ge now\_time$ 的时刻
\end{itemize}
\textbf{结论：}调度动作都在交互器断言允许域内。
\end{frame}

\begin{frame}{12. 时限把控（2 min）}
数据规模：任务最多约 1e4，进程最多 1e3，时限 8s。\\[4pt]
题解的时限策略：
\begin{itemize}
  \item 哈希表 + 向量，避免昂贵结构
  \item \texttt{nth\_element} 降低排序成本
  \item 事件后“并非每次全量重调度”：
  \begin{itemize}
    \item 前 1000 事件都调
    \item 之后每 4 次调一次
    \item 任务到达/结束事件强制调
  \end{itemize}
\end{itemize}
\textbf{工程直觉：}在效果和开销之间做自适应折中。
\end{frame}

\begin{frame}{13. 样例推演（1.5 min）}
样例：$N=2, K=4, M=16$，任务 1 与任务 2 资源冲突，不能并跑。\\[4pt]
调度：
\begin{itemize}
  \item time 0: \texttt{Schedule(1,0)}
  \item time 6: \texttt{Schedule(2,0)}
\end{itemize}
目标值：\texttt{19.1667}。\\
解释：先完整跑完 task1 再跑 task2，在该数据下优于中途抢占。

\end{frame}

\begin{frame}{14. 参数与可调方向（1.5 min）}
代码中的关键超参数：
\begin{itemize}
  \item $\theta$（短段偏好）
  \item $\lambda_{preempt}$（抢占激进程度）
  \item $\lambda_{evict,1},\lambda_{evict,2}$（驱逐损失）
  \item \texttt{mean\_segments} 的先验和 EMA 更新系数
\end{itemize}
调参建议：
\begin{itemize}
  \item 真实日志数据与理想数据可分组调参
  \item 关注抢占频率、平均回退时长、尾部任务等待
\end{itemize}
\end{frame}

\section{总结}

\begin{frame}{15. 总结与答疑（1 min）}
\textbf{一句话：}这是一个“未知总长度 + 多资源约束 + 可抢占损失”的在线调度问题。\\[6pt]
题解亮点：
\begin{itemize}
  \item 用估计段数把未知 $c_i$ 转成可决策分数
  \item 用资源紧缺度自适应平衡 GPU/内存瓶颈
  \item 用收益-代价模型控制抢占
\end{itemize}
